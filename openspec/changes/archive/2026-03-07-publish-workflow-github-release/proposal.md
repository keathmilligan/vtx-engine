## Why

The current publish workflow only pushes to crates.io but does not create a GitHub release, leaving users with no release page, changelog, or downloadable artifacts on GitHub. Adding a GitHub release as part of the publish step closes this gap and follows standard open-source practice for Rust libraries.

## What Changes

- The `publish.yml` workflow is refactored to run as two sequential jobs: `publish-crate` (crates.io) and `create-release` (GitHub release).
- The GitHub release is created automatically from the pushed tag, using the tag name as the release title.
- The release notes are auto-generated by GitHub from commit history (using `generate_release_notes: true`).
- The crate publish job continues to run on `ubuntu-latest` with the existing Linux audio dependency setup.
- The GitHub release job does not require a Rust build; it uses `softprops/action-gh-release` or the native `gh release create` approach.
- The `CARGO_REGISTRY_TOKEN` secret is already in place; a `GITHUB_TOKEN` is provided automatically by Actions and requires no new secrets.

## Capabilities

### New Capabilities

- `github-release`: Automated GitHub release creation on version tag push, with auto-generated release notes derived from commits since the previous tag.

### Modified Capabilities

- `crate-publication`: The existing crates.io publish step moves into its own job (`publish-crate`) so it can be depended upon by the release job; no behavioral change to the publish logic itself.

## Impact

- `.github/workflows/publish.yml` — restructured into two jobs.
- No changes to Rust source, `Cargo.toml`, or CI workflow (`ci.yml`).
- No new secrets required; `GITHUB_TOKEN` is implicit.
- No breaking changes.
